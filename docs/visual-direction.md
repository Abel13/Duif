# Visual Direction

DUIF should look like an illustrated postal notebook.

The interface should feel handcrafted, collectible, and tactile, as if the player is interacting with a desk full of paper cards, stamps, envelopes, old maps, animal notes, and travel records.

The app should not look like a generic website or dashboard.

## Visual Summary

The desired visual style combines:

- hand-drawn sketch lines;
- warm paper textures;
- postal stamps;
- envelopes;
- notebook pages;
- old travel maps;
- mascot profile cards;
- collectible album slots;
- ink labels;
- soft watercolor accents;
- irregular paper shapes;
- lightweight ornamental details.

The result should feel charming and polished, not messy or overly childish.

## Core Visual Fantasy

The player should feel like they are opening a personal postal field notebook.

Inside this notebook, they can:

- inspect their messenger animals;
- see travel records;
- equip accessories;
- check delivery status;
- view collected items;
- send correspondence to friends;
- browse cards, stamps, and souvenirs.

Every important UI element should feel like it belongs to this postal world.

## Color Usage

Use colors with restraint.

The background should mostly use paper tones.

Use accent colors for:

- stamps;
- level badges;
- selected tabs;
- rare item borders;
- delivery status;
- map markers;
- important actions.

Avoid large areas of saturated color.

The interface should feel calm and readable.

## Backgrounds

Avoid pure white backgrounds.

Preferred backgrounds:

- warm paper;
- aged notebook page;
- faint grid paper;
- faint map texture;
- desk-like surface;
- low-contrast paper fibers.

Textures should be subtle.

Do not use large heavy background images. Prefer small repeatable textures or CSS-generated paper effects.

## Layout Feeling

The layout should feel like arranged paper objects on a desk or notebook page.

Preferred layout patterns:

- open notebook page;
- stacked cards;
- pinned paper notes;
- album slots;
- postal forms;
- passport-style profile pages;
- folded map panels;
- envelope sections.

Avoid:

- generic dashboard cards;
- modern SaaS hero sections;
- plain white panels;
- default rectangular buttons;
- complex data tables;
- grid systems that feel too corporate.

## UI Components

### Panels

Panels should look like paper sheets, cards, notes, or album sections.

Use:

- irregular border radii;
- subtle paper background;
- ink-colored borders;
- light shadows;
- small tape corners;
- stamp-like labels;
- dashed dividers;
- handwritten-style captions where appropriate.

Panels should remain readable and structured.

### Buttons

Buttons should look like in-world objects.

Preferred button styles:

- postal stamp;
- paper tag;
- cardboard label;
- notebook tab;
- wax-like seal;
- envelope flap;
- ticket stub.

Avoid default browser-looking or generic website buttons.

Primary actions can use stamp-like styling.

Secondary actions can use paper tags or outlined labels.

Use real HTML button elements for accessibility, but style them to match the game world.

### Navigation

Navigation should feel like paper tabs or notebook sections.

Possible navigation labels:

- Ninho;
- Cartas;
- Mapa;
- Amigos;
- Loja.

The bottom navigation should not look like a default mobile app tab bar.

It should feel like tabs attached to a notebook page.

### Cards

Cards should feel collectible.

Possible card types:

- mascot card;
- equipment card;
- stamp card;
- received letter card;
- friend card;
- route card;
- reward card.

Use consistent structure:

- label or category;
- main visual;
- name;
- status or rarity;
- small metadata.

### Inputs

Inputs should feel like postal forms or notebook fields.

Use:

- paper backgrounds;
- ink borders;
- label text above the field;
- subtle ruled-line effects;
- focus states that look like ink emphasis.

Avoid default browser input styling.

## Mascot Visual Style

Mascots are the emotional center of the product.

They should look:

- hand-drawn;
- expressive;
- charming;
- readable at small sizes;
- not overly detailed;
- not hyper-realistic;
- not overly childish.

Recommended style:

- sketchy ink outline;
- soft watercolor or flat muted fills;
- simple shapes;
- expressive posture;
- visible accessories;
- clear silhouette.

Mascots should work in:

- large portrait view;
- small sidebar cards;
- map marker form;
- profile preview;
- inventory/customization preview.

## Mascot Portrait Area

The main mascot area should feel like an illustrated character card or field observation sheet.

It may include:

- mascot portrait;
- name;
- species;
- level stamp;
- current status label;
- small handwritten note;
- background wash;
- tape or corner pins;
- decorative postal marks.

The mascot should be visually dominant but not cover important gameplay information.

## Equipment and Cosmetics

Equipment should feel like collectible paper-doll accessories or inventory cards.

Examples:

- delivery satchel;
- scarf;
- cap;
- badge;
- travel goggles;
- feather charm;
- postal bag;
- route compass.

Each equipment item should have:

- small illustration or placeholder;
- name;
- type;
- rarity;
- equipped state.

Rarity should be shown with tasteful visual treatment, not aggressive colors.

Possible rarity treatments:

- border detail;
- paper color;
- stamp mark;
- metallic accent;
- small label.

## Maps

Maps should look illustrated rather than realistic.

For early versions, prefer:

- SVG world map;
- simplified continents;
- dashed route lines;
- paper-colored oceans;
- inked coastlines;
- origin/destination pins;
- tiny mascot marker;
- postal route labels.

Avoid using real map tiles in the first visual prototype.

The map should support the fantasy of travel without looking like a logistics dashboard.

## Routes

Routes should feel like hand-drawn travel paths.

Use:

- dashed lines;
- curved arcs;
- small checkpoints;
- stamp marks;
- subtle motion along the path;
- progress marker.

Do not over-animate routes.

Animation should be calm and readable.

## Icons

Icons should use a simple sketch-like line style.

Preferred icon qualities:

- rounded ends;
- slight irregularity;
- clear silhouettes;
- consistent stroke width;
- not too detailed;
- readable at small sizes.

Use SVG for simple icons.

Avoid importing a generic icon set without customization.

## Texture and Detail

Use texture to add charm, not noise.

Good details:

- faint paper grain;
- low-opacity stains;
- ink dividers;
- tape corners;
- postal cancellation marks;
- small registration numbers;
- notebook lines;
- worn card edges.

Avoid:

- high-contrast grunge;
- excessive stains;
- noisy backgrounds behind text;
- heavy shadows;
- blur-heavy effects;
- overly complex decorative frames.

## Typography

Use system fonts initially.

The first implementation should not depend on external fonts.

Later, the product may use:

- one readable body font;
- one expressive hand-drawn or postal display font.

Typography should remain readable on mobile.

Use decorative typography only for:

- titles;
- labels;
- stamps;
- badges;
- small flavor text.

Do not use decorative fonts for long paragraphs or dense information.

## Animation Direction

Animations should feel gentle and physical.

Good animation uses:

- small bounces;
- paper slide-in;
- stamp press;
- card lift;
- subtle mascot idle movement;
- route marker movement;
- envelope opening.

Animation should use mostly:

- transform;
- opacity.

Avoid animating:

- layout-heavy properties;
- large shadows;
- blur;
- filters;
- complex SVG effects.

Animations should never make the interface feel slow.

## Interaction Feel

Interactions should feel tactile.

Examples:

- pressing a stamp button slightly pushes it into the paper;
- selecting a tab lifts it forward;
- hovering a card raises it like a paper object;
- opening a panel feels like unfolding a note;
- collecting a reward feels like opening an envelope.

All interactive elements must remain clear and accessible.

## Accessibility

The visual style should not harm usability.

Requirements:

- clear text contrast;
- readable font sizes;
- visible focus states;
- real button elements;
- semantic HTML;
- no information communicated by color alone;
- comfortable tap targets;
- reduced motion support where appropriate.

Sketch style should never make the UI harder to use.

## Responsive Direction

The interface should be designed mobile first.

Mobile web is the primary experience for DUIF.

Start every screen from the narrow mobile layout, then enhance for tablet and desktop.

Desktop may feel like an open notebook with side panels, but it should not drive the base design.

Tablet may use a horizontal mascot selector.

Mobile should use a single-column layout.

The mascot portrait should appear near the top on mobile.

Navigation should remain reachable.

Primary actions should remain easy to tap with one hand when practical.

Avoid horizontal overflow.

## What the First Mascot Screen Should Feel Like

The mascot screen should feel like opening a page from a postal field notebook.

It should include:

- a paper page background;
- mascot selector cards;
- a large mascot portrait;
- level and XP as stamps or labels;
- attributes as field notes;
- equipment as collectible cards;
- current delivery as a route note;
- skills as small paper tags;
- bottom navigation as notebook tabs.

The screen should feel polished even with placeholder art.

## Visual Priorities

Prioritize in this order:

1. Readability.
2. Strong postal notebook identity.
3. Mascot emotional appeal.
4. Lightweight implementation.
5. Responsiveness.
6. Decorative detail.

Do not sacrifice readability for sketch effects.

## Things to Avoid

Avoid:

- emojis;
- default blue buttons;
- generic card UI;
- Bootstrap-like layouts;
- dashboard styling;
- glassmorphism;
- neon gradients;
- excessive shadows;
- excessive filters;
- full-screen image UI;
- overly realistic map tiles;
- cluttered decoration;
- childish cartoon overload;
- unreadable handwritten text.

## Implementation Notes

The visual style should be implemented with lightweight web techniques.

Use:

- HTML for structure;
- CSS Modules for component styling;
- CSS variables for theme;
- SVG for simple marks and icons;
- optimized images later for mascot art;
- small repeatable textures;
- transform and opacity for animations.

Do not build the UI as one large illustration.

The app should feel illustrated while remaining maintainable, responsive, and dynamic.

## Success Criteria

The visual direction is successful if:

- the app does not look like a generic website;
- the mascot feels like a collectible companion;
- the postal theme is immediately clear;
- the interface is readable on mobile;
- the style can scale to inventory, map, friends, and shop screens;
- the implementation remains lightweight;
- buttons and panels feel like objects from the game world.
